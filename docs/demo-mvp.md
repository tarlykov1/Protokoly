# Demo MVP

Демоверсия показывает полный автономный сценарий управления протоколом без подключения к Bitrix24 и без внешних API.

## Сценарий
1. Открыть приложение.
2. Перейти в «Демонстрация» и выбрать демонстрационный проект.
3. Скачать или загрузить DOCX.
4. Проверить распознанные разделы и поручения.
5. Исправить формулировки, сроки и исполнителей.
6. Запустить массовую проверку и локальную RuleBased AI-оценку.
7. Сформировать план задач.
8. Создать тестовые задачи через FakeTaskGateway.
9. Открыть результат и журнал тестовых публикаций.
10. Повторить только неуспешные операции или сбросить demo-данные.

## Docker
```bash
docker compose up --build
python -m app.cli.seed_demo
python -m app.cli.generate_demo_docx
```

## Быстрый запуск
```bash
make demo
```
Команда применяет миграции, создаёт demo-данные, генерирует DOCX и выводит URL.

## SQLite
```bash
DATABASE_URL=sqlite:///./demo.db DEMO_MODE=true alembic upgrade head
DATABASE_URL=sqlite:///./demo.db DEMO_MODE=true python -m app.cli.seed_demo
DATABASE_URL=sqlite:///./demo.db DEMO_MODE=true python -m app.cli.generate_demo_docx
```

## Данные
`python -m app.cli.seed_demo` идемпотентно создаёт демонстрационный проект, сотрудников, списки, протокол, разделы и поручения. `python -m app.cli.reset_demo` удаляет только записи с demo-ключами и не затрагивает пользовательские данные.

## Ограничения
Реальный BitrixTaskGateway, OAuth, webhook и REST-вызовы не реализованы. FakeTaskGateway создаёт стабильные simulated_external_id, сохраняет payload и возвращает `reused` при повторном external_key.

## Конфиденциальность
В DEMO_MODE данные не отправляются во внешние системы. AI-оценка выполняется локальным RuleBased provider.

## Сценарий показа руководству

Демонстрация рассчитана на 5–7 минут и проходит без терминала. Оператор открывает `/demo`, нажимает «Подготовить демонстрационные данные» и запускает guided demo.

### Рекомендуемый порядок экранов

1. `/` — объяснить проблему: протоколы превращаются в управляемый перечень поручений.
2. `/demo` — показать, что demo-данные и DOCX готовятся через UI.
3. `/demo/guided` — пройти пошаговый сценарий.
4. `/protocols/{id}` — показать карточку протокола, метрики ошибок и поручения.
5. `/protocol-tasks/{id}/edit` — показать бизнес-редактирование поручения.
6. `/protocols/{id}/publication-plan` — показать будущую структуру задач.
7. `/publication-runs` и `/demo/complete` — показать итог тестовой публикации.

### Что говорить на каждом шаге

- Загрузка: «Система принимает DOCX и показывает предпросмотр до сохранения».
- Распознавание: «Поручения, сроки и исполнители выделяются автоматически, спорные места подсвечиваются».
- Проверка: «Оператор видит ошибки до публикации, а не после создания задач».
- Редактирование: «Поля названы бизнес-языком, технические идентификаторы скрыты».
- План задач: «До интеграции видно, какие корневые задачи, подзадачи и самостоятельные задачи будут подготовлены».
- Тестовая публикация: «Это имитация; задачи в Bitrix24 не создаются».

### Подготовка demo-данных через UI

На странице `/demo` доступны кнопки «Подготовить демонстрационные данные», «Сбросить демонстрационные данные» и «Скачать демонстрационный DOCX». Seed идемпотентен, reset удаляет только demo-проект и связанные demo-сущности.

### Ограничения

- Реальная интеграция Bitrix24, OAuth, webhooks и REST-вызовы не подключены.
- Публикация является тестовой и использует fake gateway.
- AI-оценка локальная и предназначена для демонстрации качества формулировок.

### Checklist перед презентацией

- Включить `DEMO_MODE=true`.
- Выполнить миграции до актуальной головы.
- Открыть `/demo` и подготовить demo-данные.
- Проверить `/demo/guided`, `/demo/dashboard`, карточку протокола и план задач.
- Убедиться, что на экранах публикации есть фраза «Задачи в Bitrix24 не создавались».
